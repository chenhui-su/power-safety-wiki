## 二、先下载仓库压缩包并解压到本地

即使你没有仓库写权限，也建议先下载仓库压缩包并解压到本地，再开始写作和预览。这样可以确保你使用的是统一的：

- `mkdocs.yml` 站点配置
- `requirements.txt` 依赖版本
- `scripts/start.bat` 和 `scripts/start.sh` 启动脚本
- `docs/incidents/_template.md` 最新模板

推荐操作方式：

1. 打开仓库主页：https://gitee.com/ch_su/power-safety-wiki
2. 点击“克隆/下载”，再点击“下载 ZIP”
3. 将压缩包解压到本地，例如：`D:\Projects\power-safety-wiki`
4. 进入解压后的项目目录，再进行后续写作和预览

> 对内容策划组同学来说，这一步不需要安装 Git，直接下载压缩包即可。

如果你已经下载并解压过仓库，开始新任务前建议先确认自己使用的是最新压缩包版本。

## 三、按模板撰写故障分析文档

### 3.1 模板与目录

- 模板文件：`docs/incidents/_template.md`
- 案例目录建议格式：`年份-地区关键词`（例如：`2012-india`）
- 主文件统一命名：`index.md`

### 3.2 必填模块

- 顶部信息卡（发生时间、影响范围、影响人数、停电时长、损失容量、历史记录）
- 事故概述（背景、规模、核心结论）
- 事故根源分析（直接原因 + 深层原因：供需/技术/管理）
- 事故时序（关键节点，建议至少一段完整时序）
- 影响与后果（受影响规模表 + 次生影响）
- 电力系统结构性问题（系统性矛盾提炼）
- 事后改革（主要改革方向）
- 电力系统安全分析视角（知识点映射表）
- AI 辅助分析视频（必填，按模板保留嵌入区块并提交视频文件）
- 参考资料（至少 2 条，优先官方报告，建议补充学术来源）

### 3.3 写作建议

- 先列清晰时间线，再写因果分析
- 所有关键数据尽量给来源
- 不直接复制原文，注意归纳与改写
- 术语尽量统一（如机组、线路、负荷、联络线）

## 四、本地预览和截图（建议进行）

本步骤不是硬性要求，但建议在提交前进行本地预览和截图，便于自查和审核沟通。

### 4.1 一键启动（推荐）

如果你不想逐条敲命令，可直接在仓库根目录运行：

```bash
# Windows
scripts\start.bat

# macOS / Linux
chmod +x scripts/start.sh && ./scripts/start.sh
```

脚本会自动检查虚拟环境、安装依赖，并启动本地预览服务。

### 4.2 手动启动（首次配置也适用）

如果你希望自己逐步执行，可以按下面操作：

```bash
# 进入仓库根目录
cd power-safety-wiki

# 创建虚拟环境（首次一次即可）
python -m venv .venv

# 激活虚拟环境
# Windows PowerShell
.\.venv\Scripts\Activate.ps1
# Windows CMD
.\.venv\Scripts\activate.bat
# macOS / Linux
source .venv/bin/activate

# 安装依赖（首次一次即可）
pip install -r requirements.txt

# 启动本地预览
mkdocs serve
```

> 如果你的电脑里 `python` 命令不可用，可改用 `py`（Windows）或 `python3`（macOS / Linux）。

### 4.3 访问与检查

浏览器访问：`http://localhost:8000`

如进行本地预览，建议重点检查：

- 页面标题、目录、提示框是否正常显示
- 表格、Mermaid 图、引用块是否排版正常
- 视频占位区域是否仍然保留在页面中
- 修改 `index.md` 后，页面是否能自动刷新

### 4.4 常见情况处理

- 端口 `8000` 被占用时，可执行 `mkdocs serve -a 127.0.0.1:8080`，然后访问 `http://localhost:8080`
- 预览结束后，在终端按 `Ctrl + C` 停止服务
- 如果依赖安装失败或命令无法识别，再回看 `QUICK_START.md` 的环境配置部分

### 4.5 截图建议

如果你按建议进行了本地预览和截图，建议优先截到这些内容：

- 页面顶部标题与顶部信息卡
- 关键图表、表格或时序分析区域
- AI 视频占位区域

## 五、使用 NotebookLM 生成分析视频

1. 准备你的事故资料（官方报告、论文、权威新闻）
2. 在 NotebookLM 生成分析讲解内容
3. 导出或保存视频文件（建议 3-10 分钟）
4. 确保视频内容与文档结论一致

## 六、提交给内容审核组（唯一提交通道）

请至少提交以下两项材料给内容审核组：

1. 项目文件（至少包含 `docs/incidents/<case>/index.md` 与相关素材）
2. NotebookLM 生成的视频文件（或可下载原文件）

如已按建议进行本地预览和截图，可一并附上：

- 本地预览截图（PNG/JPG）

内容审核组审核通过后，会将材料转交技术组继续上线流程：

1. 技术组视频服务器维护成员上传视频并生成视频链接
2. 技术组仓库维护成员将视频链接嵌入 MD
3. 技术组仓库维护成员完成页面加入、提交与部署
